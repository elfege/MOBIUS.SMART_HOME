Webhook Dispatcher Architecture

How Hubitat events flow from the Maker API to our applications.

1. Overview

Hubitat's Maker API supports sending events to one URL per Maker API instance. Our webhook dispatcher acts as a fan-out service: it receives events from Hubitat and forwards them to multiple downstream applications.

2. Configuration

Hubitat Maker API Setting

In Maker API app settings, set "URL to send device events to by POST" to:

http://:5050/api/webhook/event

Dispatcher Environment Variables

VariableDefaultDescription
WEBHOOK_PORT5050Port the dispatcher listens on

Current Target Configuration

WEBHOOK_TARGETS=http://smarthome-app:5000/api/webhook/event,http://tiles-app:80/api/webhook/event

Content Unwrapping

Hubitat wraps event data in a content object. The dispatcher unwraps the content field before forwarding, so downstream apps receive the event data directly:

From Hubitat (raw)

{
  "content": {
    "name": "motion",
    "value": "active",
    "displayName": "Office Motion",
    "deviceId": "23",
    "descriptionText": "...",
    "unit": null,
    "type": null,
    "data": null
  }
}

To Apps (unwrapped)

{
  "name": "motion",
  "value": "active",
  "displayName": "Office Motion",
  "deviceId": "23",
  "descriptionText": "...",
  "unit": null,
  "type": null,
  "data": null
}

Ownership

0_MOBIUS.SMART_HOME solely owns the webhook-dispatcher container. TILES removed it from its own docker-compose. Both projects' Dockerfile.dispatcher and services/webhook_dispatcher.py are hardlinks to the canonical source at ~/0_SCRIPTS/webhook_dispatcher/.

Network Connectivity

After startup, the dispatcher needs access to both Docker networks to resolve container hostnames: 

docker network connect tiles_tiles-net webhook-dispatcher

This allows the dispatcher to resolve tiles-app via Docker DNS. smarthome-app is already resolvable since the dispatcher runs on the same docker-compose network.

5. SSE Considerations

Apps receiving events via webhook that use Server-Sent Events (SSE) to push to browser clients must run single-worker uvicorn.

Why single worker? Multiple uvicorn workers have isolated memory. An SSE client connected to worker A won't see events from a webhook hitting worker B. Uvicorn's async event loop handles concurrency within a single worker.

6. Reliability Notes

  • Hubitat's POST is best-effort — no retry on failure
  • The dispatcher forwards in parallel (non-blocking) to all targets
  • If a target is down, the dispatcher logs the error but continues for other targets
  • For critical use cases, combine webhook push with periodic polling as a fallback
  • Hubitat sends events for all authorized devices in the Maker API instance — apps must filter by device ID
