Dual-Command Flow: Matter + Hubitat

0_MOBIUS.SMART_HOME — February 23, 2026

1. Overview

When a device is commissioned to the local Matter fabric, MOBIUS sends commands via both protocols in parallel:

  1. Matter first (async, fire-and-forget) — lowest latency, local LAN
  2. Hubitat second (full retry + verify cycle) — cloud-relay but authoritative for state verification

If a device is not commissioned to Matter, only Hubitat fires. The _fire_matter_command() method checks the device_matter_map table and silently returns if no mapping exists.

Why Matter first? Matter commands traverse the local LAN directly (device ← Matter Server ← MOBIUS). Hubitat commands go Hub ← cloud relay ← MOBIUS, adding 200-500ms. For user-perceived responsiveness, firing Matter first makes the device react sooner.

4. UPDATING Guard (Phantom Override Prevention)

The UPDATING guard in _handle_switch() prevents the memoization system from tagging state changes as "manual override" when they're actually echoes of our own commands.

The Problem Without the Guard

Hub->>WH: Webhook: switch=on WH->>App: DeviceEvent(switch, "on") Note over App: WITHOUT guard:
App hasn't sent Hubitat cmd yet!
Thinks this is manual override! Note over App: INCORRECTLY memoizes
source='manual'
Race condition: Matter changes the device before Hubitat sends its command. The Hubitat webhook for the state change arrives before the Hubitat command completes. Without the guard, _handle_switch() interprets this as a manual user override and memoizes it, blocking future automation commands for that device.

Timing window: The UPDATING status is set at the very start of _execute_command_sync() (before sending any command) and cleared when verification completes (VERIFIED) or fails (FAILED/TIMEOUT). This covers the entire window where echoed webhooks could arrive.

6. Memoization Safety

Memoization updates are always gated on Hubitat verification, never on Matter success. This means:

  • The memo system records what Hubitat verified the device state to be after the command cycle completes.
  • Matter's fire-and-forget result is invisible to the memo system.
  • If Matter changes the device and Hubitat's verify confirms it, the memo correctly reflects the new state — it doesn't matter which protocol caused the change.
  • The UPDATING guard prevents webhooks (from either protocol) from being misinterpreted as manual overrides during the command cycle.
