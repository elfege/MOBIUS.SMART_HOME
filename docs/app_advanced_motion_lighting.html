Advanced Motion Lighting - App Architecture

App Overview: Advanced Motion Lighting Management (v2.0.0) is the first app ported from Hubitat Groovy to the 0_SMART_HOME platform. It provides motion-triggered lighting automation with memoization (detects and respects user manual overrides), health monitoring for motion sensors, illuminance-based exception handling, color/dimming control, mode-aware timeouts, and pause/resume via physical buttons. Each instance operates independently with its own device set, settings, and state.

Level 1: High-Level Automation Flow

Core Automation Cycle: Motion sensors detect movement and send events via Hubitat. The app turns on configured switches (with optional dimming and color control), starts a timeout timer, and waits. If motion re-triggers, the timeout resets. When the timeout expires with no motion, lights turn off. Throughout this cycle, the app monitors for user manual overrides (memoization) and respects them by not fighting the user's control.

Level 3: Memoization System

Purpose: Memoization prevents the app from "fighting" the user. When a user manually turns a switch on or off (detected by timing), the app remembers this override and stops controlling that specific device until conditions reset (mode change, or new motion cycle). This is the key differentiator from simple motion-triggered automations.

Memoization Rules

  • Detection Heuristic: If a switch event arrives more than 3 seconds after the app's last command to that device, it's a user manual action
  • Scope: Per-device — memoizing switch 456 doesn't affect switch 457
  • Stored Values: Switch state ("on"/"off") and dim level per device
  • Persistence: Saved to app_instances.memoization_state (JSONB) via PostgREST
  • Reset Triggers: Location mode change clears all memoization for the instance
  • Edge Case: Very fast manual interactions (under 3s) may be misclassified as app echoes

Level 5: Timeout & Scheduling

Timeout Pattern: When motion becomes inactive, a timeout is scheduled. If motion re-triggers before the timeout, the timer resets (reschedule, not create new). When the timeout fires, the app checks if any sensor is still active — if so, it reschedules. Only when all sensors report inactive does it turn off lights. This prevents flickering and handles multi-sensor scenarios.

Timeout Configuration

  • noMotionTime: The timeout value (integer, default varies by mode)
  • timeUnit: "seconds" or "minutes" — multiplier for the timeout
  • Calculation: noMotionTime * (60 if timeUnit == "minutes" else 1)
  • Reschedule vs. New: reschedule_timeout() updates the existing job in APScheduler rather than canceling and creating a new one
  • Job ID Format: timeout_{instance_id} — one timeout per instance, always overwritten
  • Persistence: Jobs are backed up to scheduled_jobs table; restored on app restart (skipping missed ones)

Level 6: Sensor Health Monitoring

Problem: Motion sensors can silently fail (battery dies, wireless interference, physical obstruction). Without health monitoring, the app would never receive motion events and lights would stay off (or on) forever.

Solution: A recurring health check job runs every 10 minutes. It tracks which sensors have reported events recently. If a sensor hasn't reported in the check interval, it's marked as non-functional. If all sensors fail and considerActiveWhenFail is enabled, the app assumes motion is active (fail-safe: lights stay on).

Health Check Details

  • Interval: Every 10 minutes (recurring APScheduler job)
  • Job ID: health_check_{instance_id}
  • Tracking: _functional_sensors dict in RuntimeInstanceState (in-memory, not persisted)
  • Fail-Safe Behavior: Configurable per-instance via considerActiveWhenFail setting (default: false)
  • Reset on Event: Any event from a sensor immediately marks it functional — health check only downgrades
  • Startup: All sensors assumed functional until first health check runs

Level 10: Location Mode Change Handling

Mode Changes: Hubitat location modes (Day, Night, Away, etc.) affect automation behavior. When the mode changes, all running instances are notified. The Advanced Motion Lighting app resets memoization (user overrides no longer apply in the new context) and re-evaluates its state from scratch. Future enhancement: mode-specific timeout values and dim levels.

App Architecture Summary

Advanced Motion Lighting Key Characteristics

  • Memoization-First Design: The app always checks for user manual overrides before controlling any device — this is the core differentiator from basic motion automations
  • Multi-Sensor Support: Multiple motion sensors per instance, with independent health tracking and functional status per sensor
  • Timeout Rescheduling: Motion re-triggers reset the timer rather than creating new timers, preventing light flickering
  • Health Monitoring: 10-minute recurring checks detect silent sensor failures with configurable fail-safe behavior
  • 5 Device Categories: Motion sensors (required), switches (required), illuminance sensor, pause buttons, contact sensors
  • Rich Light Control: On/off, dimming (0-100%), color temperature presets (2700-6500K), RGB presets, custom Kelvin
  • Pause/Resume: Physical button toggle or API call, with optional auto-resume via scheduled job
  • Mode-Aware: Memoization resets on mode change; future: mode-specific timeouts and dim levels
  • 413 Lines of Logic: Compact but feature-rich — ported from the Hubitat Groovy original

Known Limitations & Future Work

  • Mode-Specific Timeouts: Not yet implemented — currently uses single timeout value regardless of mode
  • Mode-Specific Dim Levels: Not yet implemented — same dim level across all modes
  • Memoization Timing Edge Case: The 3-second heuristic may misclassify very fast manual interactions as app echoes
  • No Occupancy Sensor Support: Currently only handles motion sensors; occupancy sensors (which distinguish presence vs. motion) are not yet supported
  • Health Check Not Persistent: _functional_sensors is in-memory only (RuntimeInstanceState); resets on restart
  • Single Illuminance Sensor: Only one lux sensor per instance; no averaging across multiple sensors
