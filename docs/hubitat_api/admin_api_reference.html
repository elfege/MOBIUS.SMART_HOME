Hubitat Admin API Reference

Compiled June 2026 against Hubitat firmware 2.5.0.143 (C-8). Endpoints reverse-engineered from the hub’s own Vue UI (/ui2/js/vue-hub2.min.js) and confirmed against live hubs in the MOBIUS.SMART_HOME fleet.

The admin API is what Hubitat’s own web UI uses to talk to the hub. It is undocumented but stable, and used in the community for years. In MOBIUS.SMART_HOME it has replaced the Maker API as the primary transport for both outbound commands and device polling. Maker API is supported as an opt-in fallback.

Why Admin over Maker

ConcernMaker APIAdmin API
Per-device opt-in Every device must be explicitly enabled in the Maker app on the hub. New devices silently invisible until you remember to add them. None. All devices visible to all callers as long as they can authenticate (or the hub is open).
Event forwarding firmware regression risk Maker silently de-arms event forwarding for individual devices across firmware updates and Maker re-saves (witnessed 2026-05-16 on the Living motion sensors). Events come via the eventsocket WebSocket, not Maker. No silent opt-out.
Device richness {id, label, attributes:[{name,currentValue}]} Full device dict: preferences, driver name, mesh status, last-activity, parent links, room assignment, capabilities, commands, dashboards using it, apps using it.
Round-trip latency Maker handler is a Groovy app processing each request — adds 50–250 ms over the wire. Hits the hub’s internal request pipeline directly — closer to web-UI latency.
Token in URL Every request includes the access token in the query string — leaked in logs and process listings. Session cookie after a one-shot POST /login. No per-request secret in the URL.

Base URL

http://[HUB_IP]/[ENDPOINT]

No path prefix, no port qualifier, no token. The hub serves the admin endpoints from the root on port 80. HTTPS is not enabled on the hub by default.

Authentication

Two modes — the client probes for the second only after a 401:

  1. Open hub. If the hub has no login password set under Settings › Hub Login Security, requests work unauthenticated. This is the operator’s default state today.
  2. Password hub. The web UI authenticates via a form POST to /login. The server sets a session cookie; subsequent requests include it. The client persists the cookie in a requests.Session and re-logins automatically on 401 or 302→/login.

POST /login

Header / fieldValue
Content-Typeapplication/x-www-form-urlencoded
usernameAdmin username
passwordAdmin password
submitLogin (sentinel the form expects)

Responses:

  • 302 Found with Set-Cookie: JSESSIONID=… — success. The redirect target is /; the client must NOT follow it (callers pass allow_redirects=False) or the cookie is overwritten before it can be captured.
  • 200 OK re-rendering the login form — bad credentials. Some firmware versions still set a cookie here; it is not valid.

Credential resolution (this codebase)

The client resolves (username, password) in this order:

  1. hub_config.admin_username + hub_config.admin_password — populated by the operator via the /hubs settings page.
  2. Environment variables HUBITAT_ADMIN_USER_<n> / HUBITAT_ADMIN_PASSWORD_<n> where <n> = hub_config.admin_creds_index — populated by pull_aws_secrets HUBITAT at boot.
  3. (None, None) — correct for an open hub (default state).

Empty strings are treated as “no credential” on both layers; pull_aws_secrets exports every key in the secret including empties, and the bash equivalent uses [[ -n "$VAR" ]].

GET /device/fullJson/<id>

Full state for a single device — preferences, driver, every command, every attribute’s current value, apps consuming it, and dashboard widgets using it. Pricier per call than the Maker bulk endpoint, but ours-style code only fetches the devices we actually need state for.

Use case: command verification, capability discovery, the reconcile poll’s per-subscribed-device sweep.

Response shape (top-level keys we read):

    "label": "Fan Bathroom",
    "displayName": "Fan Bathroom",
    "name": "GE Smart Fan Control",

    "currentStates": {
      "switch": { "name": "switch", "value": "on", "stringValue": "on", "dataType": "ENUM" },
      "level":  { "name": "level",  "value": 65,   "stringValue": "65", "dataType": "NUMBER" },
      "speed":  { "name": "speed",  "value": "medium", "dataType": "ENUM" }
    },
    "commands": [
      { "name": "on",        "capability": true, "arguments": null, "parameters": null },
      { "name": "off",       "capability": true, "arguments": null, "parameters": null },
      { "name": "setLevel",  "capability": true, "arguments": null, "parameters": [{ "type": "NUMBER", "name": "level" }] },
      …
    ],
    "capabilities": ["Switch", "SwitchLevel", "FanControl", "Refresh", "Configuration"],
    "preferences": { … },

      { "id": 1530, "label": "HUMIDITY MANAGEMENT BATHROOM", "name": "HUMIDITY MANAGEMENT", "disabled": true },

Shape gotcha: currentStates here is a dict keyed by attribute name. The Maker API returns it as a list [{name, currentValue}, …]. The codebase calls to_maker_shape() to normalize, otherwise downstream extract_attribute() finds nothing and command verification silently breaks (live-fixed 2026-05-26 after the firmware contract flip).

POST /device/runmethod — command execution

The endpoint that fires commands. Two firmware contracts in the wild — the client tries the modern one first and falls back to the legacy one on transport rejection.

Current contract (firmware 2.5.0.x JSON)

Header / fieldValue
Content-Typeapplication/json
Body{"id": <device_id>, "method": "<command>", "args": […]}

The args array is a list of typed entries:

// no argument
"args": []

// numeric argument (e.g. setLevel(50))
"args": [{ "type": "NUMBER", "value": 50 }]

// string argument
"args": [{ "type": "STRING", "value": "medium" }]

The client’s _build_runmethod_args() infers the type from the value: integral numerics become NUMBER/int, fractional numerics become NUMBER/float, anything non-numeric becomes STRING. Hubitat coerces within reason but sending the right type avoids parameter-binding surprises.

Successful response: HTTP 200 with JSON body {"success": true} (or a richer envelope — we only check success).

Command failure (the device rejected it, e.g. unknown command, bad arg type): HTTP 200 with {"success": false}. Not a transport failure — the client returns (ok=False, transport_failed=False) and does NOT retry with the legacy contract.

Legacy contract (older firmware, form-urlencoded)

Header / fieldValue
Content-Typeapplication/x-www-form-urlencoded
idDevice ID
methodCommand name
arg + valueOptional argument; same value goes into both fields

Used only as a fallback. A hub on the JSON contract rejects this with a transport error and the client doesn’t retry beyond one attempt per contract.

Contract flip detection (the 2026-05-26 incident)

Firmware 2.5.0.x changed this endpoint from form-urlencoded to JSON with no backward compatibility. A hub that receives the wrong body type answers with a bare HTTP 500 "Server error" — not a 415 — so the surface symptom is “every command fails verification with got=None” rather than an obvious content-type error.

Our send_command() tries JSON first, sees a 4xx/5xx, retries form-urlencoded once, and records which one worked. The contract-watch service probes a no-arg refresh on a known device periodically and writes the verdict to hub_health so the dashboard surfaces a contract flip the moment it happens.

Response interpretation table

HTTPBodyVerdictAction
200–399{"success": true} (or no JSON envelope)OKnone
200–399{"success": false}Command failurelog, do NOT fall back
≥ 400anyTransport failure (wrong contract)fall back to the other encoding once

Fields we use:

  • version — firmware version. The contract-watch service stamps this on hub_health.platform_version on every check.
  • alerts.* — the same flags the hub’s own dashboard shows. Worth surfacing in a future hub-health widget.

GET /device/updateLabel

Rename a device on the hub. Used by device_name_normalizer for the strip-trailing-“on <hub>” pass on Hub-Mesh linked labels.

Endpoint (verbatim, from the hub’s own Vue UI):

fetch("/device/updateLabel?" + new URLSearchParams({deviceId, label}))
  .then(r => r.json())
  .then(r => { r === true && cb() })

A GET with query params deviceId and label. The hub replies with the JSON literal true on success. Same call the web UI fires on a manual rename, so it persists exactly like one.

Note: takes the device’s per-hub Hubitat id, not our canonical PK.

Eventsocket (WebSocket event stream)

Not strictly part of the admin REST API, but the same endpoint family — the hub exposes a raw WebSocket at ws://<HUB_IP>/eventsocket that emits every device event the hub processes, regardless of any per-device Maker-API opt-in.

Replaces: the Maker-API HTTP webhook intake. Maker webhook is the path that silently de-armed itself for individual devices in the 2026-05-16 Living-motion incident. The eventsocket has no per-device opt-in to lose.

Frame shape (real, switch event):

{
  "source":  "DEVICE",
  "name":    "switch",
  "value":   "on",
  "type":    "physical",
  "unit":    null,
  "data":    null,
  "deviceId": "2781",
  "displayName": "Fan Bathroom",
  "descriptionText": null
}

Two source types we consume:

  • DEVICE — per-device state events. Routed to subscribed app instances via the webhook router.
  • LOCATION — hub-level events. Only the mode events are routed (to route_mode_change). Sunset/sunrise/etc. are skipped.

Everything else (APP_STATUS, HUB_INFO) is dropped at the WS drain.

physical vs digital: the type field distinguishes events caused by physical interaction (physical) from events caused by a command we (or another app) issued (digital). Apps subscribing to the same device’s switch event must NOT re-issue a command on every event — doing so creates a self-feedback loop at the hub roundtrip rate (the 2026-06-05 Fan Bathroom storm). The framework now skips auto-subscription for pure-output categories; see services/instance_manager.py::category_events.

Reliability: the eventsocket WS does not heartbeat the way the hub UI would — ws.open == True is not proof events are flowing. Our client adds a 120 s data-flow watchdog that recycles the connection if no event arrives. Reset of the exponential backoff happens on first data, not on first connect.

Defensive layers in this codebase

Circuit breaker per hub

Every HubitatAdminClient wraps every request through a per-hub circuit breaker (hubitat:<hub_ip>). Trips after 5 consecutive request failures within 60 s, cools down 30 s before letting a probe through. While OPEN, the client raises CircuitBreakerOpen instead of hitting the hub — callers fall back to cached state or surface the degradation. Observability: GET /api/health/breakers on this app.

Key rules:

  • Any exception from the underlying requests call counts as a failure.
  • HTTP 5xx is converted to an exception by the client wrapper so the breaker counts it.
  • A 401-then-re-login does NOT count — only the outer call’s final outcome matters.

Contract-watch heartbeat

The services/hub_contract_watch.py service polls /hub2/hubData (for the firmware version) and calls HubitatAdminClient.probe_command_path() on every hub on a schedule. Writes the verdict to hub_health:

ColumnMeaning
platform_versione.g. "2.5.0.143"
command_path_oktrue / false — did SOME contract accept a no-arg refresh?
command_path_contract"json" / "form" / "none"
command_path_checked_atTimestamp of the last probe
command_path_errorBody snippet from the last failing call (for diagnosis)

The hubs page surfaces this as a green / amber / red badge on each hub card.

Hub-Mesh suffix & dedup

Hub Mesh appends “ on <hub_name>” to name on every linked-device row. Two consequences:

  1. The same physical device shows up once per hub that has it shared (1 native + N mirrors). The device_to_hubs_classifier uses linkedDevice to drop mirrors before they reach the canonical devices table.
  2. The label as the user sees it ends up with the suffix on the mirrors. device_name_normalizer (default-off) runs a dry-run-first pass that strips the suffix via GET /device/updateLabel. Data-driven from /hub2/hubData.locationName, not hardcoded.

Python usage

Get the per-hub client and call its public methods:

from services.hubitat_admin_client import get_client, to_maker_shape

client = get_client(hub_ip="", hub_name="home_2")

# List every device the hub knows
devices = client.get_all_devices()                       # list[dict]

# Full state for one device
raw = client.get_device(device_id=2781)                  # dict | None
maker_shaped = to_maker_shape(raw)
# -> {"id": "2781", "label": "Fan Bathroom",
#     "attributes": [{"name":"switch","currentValue":"on"}, ...]}

# Fire a command
ok = client.send_command(device_id=2781, command="setLevel",
                          argument=50)                    # bool

# Rename a device
ok = client.set_device_label(device_id=2781,
                             new_label="Bathroom Fan")    # bool

# Read the hub's location name
loc = client.get_location_name()                          # "Home 2" | None

# Health-probe the command transport
verdict = client.probe_command_path(device_id=2781)
# -> {"ok": True, "contract": "json", "status": 200, "error": None}

The client is a per-hub singleton. get_client() reuses the cached requests.Session (and therefore the auth cookie). invalidate_clients() drops all cached clients after a credentials rotation.

Quirks & gotchas

QuirkSymptomMitigation
Successful POST /login returns 302; following the redirect overwrites the cookie Authenticated client behaves as if it didn’t log in Pass allow_redirects=False on POST /login calls.
Wrong content-type on /device/runmethod returns bare HTTP 500 Commands silently fail; verification reports got=None Two-contract retry in send_command() + contract-watch alerting.
currentStates shape differs from Maker (dict vs list) Downstream extract_attribute() finds nothing; every verify fails Always pass get_device() output through to_maker_shape() before reading attributes.
Hub-Mesh adds “ on <hub_name>” to mirror device names Same device shows up multiple times under nearly identical labels; pickers get noisy; instances accidentally bind to a mirror Classifier uses linkedDevice for dedup; device_name_normalizer for label cleanup.
Eventsocket WS does not heartbeat at the protocol level ws.open == True after silent hub-side disconnect; events stop flowing without notice 120 s data-flow watchdog in hubitat_eventsocket_client.py recycles the connection on no-event.
Eventsocket fan-out via Hub Mesh emits the same event on every hub the device is shared on Same event ingested N times; routing decisions made N times Mesh-mirror filter at ingest: drop events whose origin hub ≠ the device’s canonical hub. Key on <hub_ip>:<deviceId>.
The hub serves admin endpoints over plain HTTP; HTTPS is not on by default Cookies and admin credentials cross the LAN in cleartext LAN-only access posture today. If exposing externally: VPN or reverse-proxy with TLS termination.
