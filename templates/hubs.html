Admin API credentials — when and how

If this hub's Settings → Hub Login Security is OFF (Hubitat default), leave these fields blank — the admin API is open on the LAN and Mobius connects without auth.

If you enable Hub Login Security, Mobius needs the username/password to call POST /login, persist the session cookie, and use it for every subsequent /device/... call.

Two ways to supply them, in priority order:

  1. Inline below — type them into the Username + Password fields. Stored in the DB (plain for now; a KEK-encryption layer is on the roadmap).
  2. AWS Secrets Manager — set the eight placeholders HUBITAT_ADMIN_USER_1..4 and HUBITAT_ADMIN_PASSWORD_1..4 in your HUBITAT secret (already provisioned). Then put the matching slot number (1, 2, 3 or 4) into AWS slot for this hub. Values flow: AWS Secrets Manager → pull_aws_secrets HUBITAT in start.sh → env vars → admin client.

The inline fields win if both are populated. To switch from inline to AWS, clear the inline fields and set the slot.

Maker API — legacy fallback (default OFF)

The primary intake is the eventsocket WS (ws://hub/eventsocket). Every event from every device streams through it, no per-device opt-in. Mobius writes everything to event_log + raw_events; the database is the source of truth for state.

Outbound HTTP (reconcile poll + commands + command verify) uses Hubitat's admin API by default — POST /device/runmethod for commands, GET /device/fullJson/<id> for verify and reconcile. Faster than Maker (skips the Groovy app layer) and doesn't need per-device opt-in.

The toggle below re-enables Maker API for outbound HTTP only — a fallback if the admin API has issues on your firmware version. The WS intake is unaffected either way.

If the eventsocket itself fails (hub reboot, firmware update), the dashboard surfaces a warning. WS will reconnect on its own; nothing to do unless the failure persists, in which case the warning suggests escalating.
