Advanced Motion Lighting - App Architecture

App Overview: Advanced Motion Lighting Management (v2.0.0) is the first app ported from Hubitat Groovy to the 0_MOBIUS.SMART_HOME platform. It provides motion-triggered lighting automation with source-tracked memoization (detects and respects user manual overrides), three-tier motion detection (memory → live state → event history), health monitoring for motion sensors, illuminance-based exception handling, color/dimming control, Always-Off/On enforcement with mode gating, and pause/resume via physical buttons. Device commands go through DeviceCommander with nested retries and state verification, with optional dual-path Matter protocol. Each instance operates independently with its own device set, settings, and state.

Level 1: High-Level Automation Flow

Core Automation Cycle: Motion sensors detect movement and send events via Hubitat. The app turns on configured switches (with optional dimming and color control) via DeviceCommander (threaded execution with nested retries and state verification), starts a timeout timer, and waits. If motion re-triggers, the timeout resets. When the timeout expires, the app uses three-tier motion detection (memory → live state → event history) to verify no sensor is still active before turning off. Throughout this cycle, the app monitors for user manual overrides via source-tracked memoization and respects them. After normal light control, Always-Off/On enforcement runs to ensure designated devices stay in their forced state.

Level 3: Memoization System

Purpose: Memoization prevents the app from "fighting" the user. When a user manually changes a switch state (detected via source tracking), the app remembers this override and stops controlling that specific device until conditions reset (mode change, pause/resume, or instance update). The key innovation is the four-source tracking model and verified-only updates.

Memoization Rules

  • Verified-Only Updates: Memo is only updated when a command is verified (device state confirmed via live Hubitat API poll). Unverified commands do not change memo.
  • Override Detection: If a switch event's state contradicts the memo's expected state, it's tagged as source='manual' (user override)
  • Scope: Per-device — memoizing switch 456 doesn't affect switch 457
  • Key Format: Regular switches keyed by device name; keep devices keyed by keep:{device_id}
  • Stored Values: {state: "on"/"off", source: "app"/"manual"/"pause"/"unknown"} and dim level per device
  • Persistence: Saved to app_instances.memoization_state (JSONB) via PostgREST
  • Reset Triggers: Mode change, pause/resume, instance update — all re-seed with source='app'
  • Seeding: On init/resume, keep_off/keep_on devices are seeded with expected state and source='app'

Level 5: Timeout & Scheduling

Timeout Pattern: When motion becomes inactive, a timeout is scheduled. If motion re-triggers before the timeout, the timer resets (reschedule, not create new). When the timeout fires, the app checks if any sensor is still active — if so, it reschedules. Only when all sensors report inactive does it turn off lights. This prevents flickering and handles multi-sensor scenarios.

Timeout Configuration

  • noMotionTime: The timeout value (integer, default varies by mode)
  • timeUnit: "seconds" or "minutes" — multiplier for the timeout
  • Calculation: noMotionTime * (60 if timeUnit == "minutes" else 1)
  • Reschedule vs. New: reschedule_timeout() updates the existing job in APScheduler rather than canceling and creating a new one
  • Job ID Format: timeout_{instance_id} — one timeout per instance, always overwritten
  • Persistence: Jobs are backed up to scheduled_jobs table; restored on app restart (skipping missed ones)

Level 6: Sensor Health Monitoring

Problem: Motion sensors can silently fail (battery dies, wireless interference, physical obstruction). Without health monitoring, the app would never receive motion events and lights would stay off (or on) forever.

Solution: A recurring health check job runs every 10 minutes. It tracks which sensors have reported events recently. If a sensor hasn't reported in the check interval, it's marked as non-functional. If all sensors fail and considerActiveWhenFail is enabled, the app assumes motion is active (fail-safe: lights stay on).

Health Check Details

  • Interval: Every 10 minutes (recurring APScheduler job)
  • Job ID: health_check_{instance_id}
  • Tracking: _functional_sensors dict in RuntimeInstanceState (in-memory, not persisted)
  • Fail-Safe Behavior: Configurable per-instance via considerActiveWhenFail setting (default: false)
  • Reset on Event: Any event from a sensor immediately marks it functional — health check only downgrades
  • Startup: All sensors assumed functional until first health check runs

Level 12: Three-Tier Motion Detection

Problem: Simple motion checking (is any sensor currently "active"?) misses cases where motion was recent but the sensor transitioned to inactive, or where in-memory state was lost after restart.

Solution: _is_motion_active() uses three progressively more expensive checks, returning True as soon as any tier confirms activity.

Tier Details

  • Tier 1 (In-Memory): Fastest path (~0ms). Checks self._runtime.last_motion_time against timeout. Used for normal runtime when events flow regularly.
  • Tier 2 (Live State): Queries Hubitat API for each functional sensor's current motion attribute (~50-200ms per sensor). Catches motion that's active right now but wasn't received via webhook.
  • Tier 3 (Event History): Queries last ~20 events per sensor from Hubitat API (~200-500ms per sensor). Searches for motion=active events within the timeout window. Catches recent motion that has already transitioned to inactive.
  • Fail-Safe: If no functional sensors exist, defers to considerActiveWhenFail setting. True = keep lights on. False = allow turn off.

Level 13: Always-Off/On Enforcement

Purpose: Devices in the keep_off_switches and keep_on_switches categories are forced to their designated state after every control cycle. Enforcement respects user manual overrides (via source tracking) and can be gated to specific location modes. See Always-Off/On detailed documentation for full decision tables.

Enforcement Points

  • master(): After normal light control, calls _enforce_keep_switches()
  • initialize(): On startup, enforces keep rules immediately (does NOT call master)
  • _health_check(): Every 10 minutes, re-enforces keep rules
  • resume(): On unpause, re-seeds memo then calls master which enforces

Mode Gating

Always-Off and Always-On enforcement can be restricted to specific location modes via keepOffModes and keepOnModes settings. An empty array means "enforce in all modes" (default).

Conflict Resolution: If a device appears in both keep_off_switches and keep_on_switches, keep_off wins. The code removes conflicting IDs: keep_on_ids = keep_on_ids - keep_off_ids.

Level 14: Initialization Behavior

Critical Design Decision: initialize() does NOT call master(). If it did, _is_motion_active() would return False (no motion event received yet) and all lights would immediately turn off on startup. Instead, initialize only:

initialize() Flow

  1. Log startup with instance label
  2. Seed _functional_sensors dict from configured motion sensors
  3. Call _init_memoization_keys() — seeds keep_off/keep_on devices with expected state and source='app'
  4. Schedule health check job (every 10 minutes)
  5. Call _enforce_keep_switches() — enforce always-off/on rules immediately
  6. Schedule initial timeout run

What it does NOT do: Call master(), turn off lights, or change any regular switch state. This prevents false light-off on startup/restart.

App Architecture Summary

Advanced Motion Lighting Key Characteristics

  • Source-Tracked Memoization: Four-source model (app/manual/pause/unknown) with verified-only updates. Only confirmed device state changes update memo. User overrides are detected and respected.
  • Three-Tier Motion Detection: Memory → live state → event history. Progressively more expensive checks ensure no false "no motion" when sensors are slow or events were missed.
  • DeviceCommander Integration: All commands go through centralized commander with 8-worker ThreadPool, nested retries (network→verify→operation), and state verification via live API poll.
  • Matter Dual-Path: For mapped devices, commands are sent simultaneously via Hubitat and Matter protocol (fire-and-forget) for lower latency.
  • Always-Off/On Enforcement: Designated devices forced to their state after every control cycle, with mode gating and user override respect.
  • Multi-Sensor Support: Multiple motion sensors per instance, with independent health tracking and functional status per sensor
  • Timeout Rescheduling: Motion re-triggers reset the timer rather than creating new timers, preventing light flickering
  • Health Monitoring: 10-minute recurring checks detect silent sensor failures, with configurable fail-safe behavior and periodic keep-switch enforcement
  • 8 Device Categories: Motion sensors, switches (required). Optional: illuminance, pause buttons, contacts, pause switches, keep-off, keep-on.
  • Rich Light Control: On/off, dimming (0-100%), color temperature presets (2700-6500K), RGB presets, custom Kelvin
  • Pause/Resume: Physical button toggle (configurable event type) or API call, with optional auto-resume and pause switch control
  • Mode-Aware: Memoization resets on mode change. Always-Off/On can be gated to specific modes. Future: mode-specific timeouts and dim levels.
  • 1,240 Lines of Logic: Feature-rich — ported and significantly extended from the Hubitat Groovy original

Known Limitations & Future Work

  • Mode-Specific Timeouts: Not yet implemented — currently uses single timeout value regardless of mode
  • Mode-Specific Dim Levels: Not yet implemented — same dim level across all modes
  • No Occupancy Sensor Support: Currently only handles motion sensors; occupancy sensors (which distinguish presence vs. motion) are not yet supported
  • Health Check Not Persistent: _functional_sensors is in-memory only (RuntimeInstanceState); resets on restart
  • Single Illuminance Sensor: Only one lux sensor per instance; no averaging across multiple sensors
  • Three-Tier Cost: Tier 2 and Tier 3 motion detection require live Hubitat API calls (~50-500ms each); may be slow on congested networks
