Hubitat Maker API — Complete Reference

Compiled February 2026 from official Hubitat documentation and community sources.

1. Overview

The Maker API is a built-in HTTP-based RESTful API on Hubitat Elevation hubs. It enables local (LAN) and optional cloud device control, status queries, and event streaming without any cloud dependency for local use.

  • Token-based authentication (query-string parameter)
  • Selective device authorization (only exposed devices are accessible)
  • Optional HSM, Mode, and Hub Variable control
  • Real-time event push via HTTP POST webhooks

2. Installation & Configuration

  1. Navigate to Apps → Add Built-In App → Maker API
  2. Select devices to authorize
  3. (Optional) Select hub variables to expose
  4. (Optional) Enable "Allow control of modes"
  5. (Optional) Enable "Allow control of HSM"
  6. (Optional) Enable debug logging
  7. Click Update — the hub generates an access token and endpoint URLs
  8. Click Done

Token Management

  • Token is generated automatically on install
  • Reset via "Create New Access Token" in app settings (invalidates old token)
  • All integrations must be updated after a token change

Network Access

  • Local (LAN): enabled by default
  • Cloud: disabled by default; enable for remote access (uses Hubitat cloud relay)

Multiple Instances

You can create multiple Maker API app instances, each with its own token and device subset, for granular access control across different integrations.

3. URL Format

http://[HUB_IP]/apps/api/[APP_ID]/[ENDPOINT]?access_token=[TOKEN]
SegmentDescriptionExample
HUB_IPHub's LAN IP address
APP_IDMaker API app number (shown in app settings)268
ENDPOINTAPI path (see sections below)devices/all
TOKENAccess token generated by the appabc123-def456...

4. API Endpoints

4.1 Device Discovery

GET /devices

Returns a JSON array of authorized devices (minimal info).

[
  { "id": "1", "name": "Generic Z-Wave Switch", "label": "Living Room Light" },
  { "id": "23", "name": "Generic Zigbee Motion Sensor", "label": "Office Motion" }
]
GET /devices/all

Returns comprehensive details for all authorized devices including capabilities, attributes (current values), and available commands.

[
  {
    "id": "1",
    "name": "Generic Z-Wave Smart Switch",
    "label": "Living Room Light",
    "type": "Generic Z-Wave Smart Switch",
    "date": "2024-01-15T10:30:00+0000",
    "model": null,
    "manufacturer": null,
    "capabilities": ["Switch", "Configuration", "Refresh", "Actuator"],
    "attributes": {
      "dataType": null,
      "values": null,
      "switch": "on"
    },
    "commands": [
      { "command": "configure" },
      { "command": "off" },
      { "command": "on" },
      { "command": "refresh" }
    ]
  }
]
Performance note: On hubs with many devices, /devices/all can be slow. Consider caching results and using per-device endpoints for updates.
GET /devices/{device_id}

Returns full details for a single device (same schema as one element of /devices/all).

4.2 Device Commands

GET /devices/{device_id}/commands

Returns the list of available commands for a device.

[
  { "command": "off", "type": [] },
  { "command": "on", "type": [] },
  { "command": "setLevel", "type": ["NUMBER"] },
  { "command": "setColor", "type": ["COLOR_MAP"] }
]
GET /devices/{device_id}/{command}

Execute a command on a device. For commands that take no parameters.

Examples

/devices/1/on              # Turn device on
/devices/1/off             # Turn device off
/devices/1/lock            # Lock a lock
/devices/1/unlock          # Unlock a lock
/devices/1/open            # Open a valve/door
/devices/1/close           # Close a valve/door
/devices/1/refresh         # Refresh device state
GET /devices/{device_id}/{command}/{secondary_value}

Execute a command with one or more parameters. Multiple parameters are comma-separated.

Examples

/devices/1/setLevel/50                          # Set dimmer to 50%
/devices/1/setLevel/75,5                        # Set to 75% over 5 seconds
/devices/1/setColorTemperature/3500             # Set color temp to 3500K
/devices/1/setColor/Red                         # Named color
/devices/1/setColor/0,100,100                   # HSB: hue,saturation,level
/devices/1/setColor/%7B%22hex%22%3A%22FF0400%22%7D  # URL-encoded {"hex":"FF0400"}
/devices/1/setThermostatSetpoint/72             # Thermostat setpoint
Parameter types: integer, decimal, or string. Multiple params separated by commas.

4.3 Device Attributes

GET /devices/{device_id}/attribute/{attribute_name}

Returns the current value of a specific attribute.

{
  "id": "123",
  "attribute": "switch",
  "value": "on"
}

4.4 Device Events

GET /devices/{device_id}/events

Returns recent events for the device (approximately 20, reverse chronological).

[
  {
    "date": "2026-02-25T14:30:00+0000",
    "value": "active",
    "device_id": "23",
    "label": "Office Motion",
    "name": "motion",
    "isStateChange": true,
    "source": "DEVICE"
  },
  {
    "date": "2026-02-25T14:25:00+0000",
    "value": "inactive",
    "device_id": "23",
    "label": "Office Motion",
    "name": "motion",
    "isStateChange": true,
    "source": "DEVICE"
  }
]

4.5 Hub Modes

GET /modes

List all available modes with the currently active one flagged.

Requires: "Allow control of modes" enabled in Maker API settings.

[
  { "id": "1", "name": "Day",   "active": false },
  { "id": "2", "name": "Night", "active": true  },
  { "id": "3", "name": "Away",  "active": false }
]
GET /modes/{mode_id}

Change the hub to the specified mode.

/modes/1    # Switch to mode with id 1

4.6 Hubitat Safety Monitor (HSM)

GET /hsm

Get the current HSM status.

{ "hsm": "disarmed" }

Possible status values: disarmed, armedAway, armedHome, armedNight

GET /hsm/{command}

Arm or disarm HSM.

Requires: "Allow control of HSM" enabled in Maker API settings.

CommandDescription
armAwayArm in Away mode
armHomeArm in Home mode
armNightArm in Night mode
disarmDisarm HSM
Important: Commands use the imperative form (armAway), not the status form (armedAway). Using status names as commands will fail.

4.7 Hub Variables

GET /hubvariables

Returns all authorized hub variables.

Requires: Variables selected under "Allow endpoint to control these hub variables".

[
  { "name": "myStringVariable", "value": "testing123", "type": "string" },
  { "name": "outsideTemp", "value": "42.5", "type": "decimal" }
]
GET /hubvariables/{variable_name}

Returns the value and type of a single hub variable.

{ "name": "myStringVariable", "value": "testing123", "type": "string" }
GET /hubvariables/{variable_name}/{value}

Set the value of a hub variable. Value must be URL-encoded if it contains special characters.

4.8 Webhook URL Configuration (via API)

GET /postURL/{encoded_url}

Programmatically set the webhook POST URL for event notifications.

/postURL/http%3A%2F%2F%3A5050%2Fapi%2Fwebhook%2Fevent
GET /postURL

Clear the webhook URL (stop sending event notifications).

5. Webhook Event Format

When a webhook URL is configured (either in the Maker API settings UI or via the /postURL endpoint), Hubitat sends an HTTP POST to that URL every time a subscribed device generates an event.

5.1 Device Event Payload

{
  "content": {
    "name": "switch",
    "value": "off",
    "displayName": "Living Room Light",
    "deviceId": "1",
    "descriptionText": "Living Room Light was turned off",
    "unit": null,
    "type": null,
    "data": null
  }
}

5.2 Payload Fields

FieldTypeDescription
contentObjectWrapper object for the event data
content.nameStringAttribute name (e.g., "switch", "motion", "temperature")
content.valueStringNew attribute value (e.g., "on", "active", "72.5")
content.displayNameStringHuman-readable device label
content.deviceIdStringNumeric device ID (as string)
content.descriptionTextStringHuman-readable event description
content.unitString|nullUnit of measurement (e.g., "°F", "%", "lux")
content.typeString|nullEvent type (usually null for device events)
content.dataString|nullAdditional event data (device/driver specific)

6. Security Considerations

  • Token is the sole auth mechanism — treat it as a sensitive credential
  • No native HTTPS support — use a reverse proxy (nginx) for TLS termination
  • No HTTP header auth — token is always a query parameter
  • Restrict to LAN unless cloud access is specifically needed
  • Selective device authorization limits blast radius
  • Multiple Maker API instances allow per-integration token isolation

7. Limitations

  • No WebSocket support — push is via HTTP POST only
  • Event history limited to ~20 recent events per device
  • No access to hub configuration, logs, or non-authorized resources
  • /devices/all can be slow on large device lists
  • All endpoints use GET (even commands that mutate state)
  • Hub variables require separate authorization in Maker API settings
