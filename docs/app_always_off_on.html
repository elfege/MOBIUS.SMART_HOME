Always Off / Always On - Feature Documentation

Feature Overview: The Always Off / Always On feature allows users to designate specific switches that should be forced to a particular state regardless of motion automation. Devices in the "Always Off" list are forced off; devices in the "Always On" list are forced on. The feature uses four-source tracking (app/manual/pause/unknown) to detect and respect user manual overrides. Enforcement can be gated to specific location modes via keepOffModes and keepOnModes settings. Enforcement runs in master(), initialize(), _health_check() (every 10 min), and on mode change.

Core Concept

Simple Rule: If a device is in the always-off list, turn it off. If it's in the always-on list, turn it on. Enforcement can optionally be restricted to specific location modes via keepOffModes / keepOnModes settings (empty = all modes).

The Exception: Source-tracked memoization detects when the user has manually overridden the app's action. If the memo source is 'manual', the app respects that override. If the source is 'app' or 'pause', enforcement proceeds. Overrides reset on mode change, pause/resume, or instance update.

Conflict Resolution: If a device appears in both keep_off and keep_on categories, keep_off wins: keep_on_ids = keep_on_ids - keep_off_ids.

Always-Off Decision Table

Goal state: Device should be OFF. The app checks the memo's source field to determine whether to enforce or skip.
Memo Source Device State Interpretation Action
source='app' ON App set expected state but device is wrong FORCE OFF — app-initiated, enforce rule
source='pause' ON Pause/resume turned it on FORCE OFF — pause is app-initiated, enforce
source='manual' ON User manually turned it back on SKIP — user override, respect until reset
source='unknown' ON No memo or unknown origin SKIP — conservative, don't fight unknown state
any OFF Already off Nothing — goal state already achieved

Memoization Interaction

How memoization applies: The memoization system tracks what the APP last commanded each device to do, with a source field distinguishing the origin. Keep device memos are keyed by keep:{device_id} format for consistency. On initialization and resume, keep devices are seeded with expected state and source='app', establishing the baseline for override detection.

Exclusion from Normal Light Control

Important: Devices in the keep_off_switches and keep_on_switches categories are excluded from the normal motion-triggered _control_lights() pipeline. This prevents the motion automation from turning on a device that should always be off (or vice versa).

The exclusion happens in _control_lights():
if device_id in keep_off_ids or device_id in keep_on_ids: continue

Mode Gating Settings

Optional mode restriction: By default, enforcement runs in all location modes. The keepOffModes and keepOnModes settings allow restricting enforcement to specific modes. An empty array (default) means "enforce in all modes."
Setting Type Default Description

keepOnModes array (strings) [] Location modes where Always-On is enforced. Empty = all modes.

When Enforcement Runs

_enforce_keep_switches() is called at four points:
  • Inside master() (line 416): After normal motion light control, enforcement runs to override any devices that should be always-off or always-on, regardless of what the motion logic decided.
  • Inside initialize() (line 175): On startup, enforcement runs immediately. This ensures keep devices are in correct state from the moment the instance starts, without waiting for a motion event.
  • Inside _health_check() (line 1034): Every 10 minutes, enforcement re-runs. This catches cases where no motion events have occurred but keep devices have drifted (e.g., physically toggled, power restored).
  • Inside on_mode_change(): After memoization is reset (re-seeded with source='app') and master() runs, enforcement ensures correct state for the new mode context.

Summary

Key Characteristics

  • Four-Source Tracking: Distinguishes app, manual, pause, unknown sources. Only manual and unknown skip enforcement.
  • Mode Gating: Optional keepOffModes / keepOnModes restrict enforcement to specific location modes. Empty = all modes (default).
  • Four Enforcement Points: master(), initialize(), _health_check() (10 min), on_mode_change()
  • Memo Seeding: On init/resume/mode-change, keep devices are seeded with expected state and source='app', establishing baseline for override detection
  • Conflict Resolution: Device in both keep_off and keep_on: keep_off wins (keep_on_ids - keep_off_ids)
  • Excluded from Normal Control: Always-off/on devices are skipped by _control_lights()
  • Pause Switch Interaction: On resume, pause switches skip keep_off devices (enforcement handles them). On pause, skip keep_on devices.
  • Verified-Only Memo: Only confirmed (verified=True) commands update memo, preventing false override detection from failed commands
