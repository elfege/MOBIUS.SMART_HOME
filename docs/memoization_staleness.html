Stale Memoization: Bug Analysis & Fix

Bug Report (March 1, 2026): Lights turned off by timeout would not turn back on when motion was detected. Root cause: memoization state became stale, causing _should_skip_due_to_memo() to incorrectly skip the turn-on command.

1. Background: How Memoization Works

The memoization system tracks what the app last did to each switch. When memoize is enabled, _should_skip_due_to_memo(device_name, action) checks: "Did we already set this device to the desired state?" If yes, it skips the command to avoid redundant work.

flowchart LR subgraph Memo Dict M["switch_state: {
'Office Lamp': {state: 'on', source: 'app'}}"] end subgraph Skip Check Q{"memo_state == action?"} Q -->|"memo='on', action='on'"| SKIP["SKIP command"] Q -->|"memo='on', action='off'"| PROCEED["PROCEED with command"] Q -->|"memo='off', action='on'"| PROCEED2["PROCEED with command"] end M --> Q
Key rule: Memo is only updated when a command is verified (device state confirmed via live Hubitat API poll after the command). Unverified commands do NOT update memo — this is intentional safety, but it creates the staleness vector.

Source: apps/advanced_motion_lighting/app_logic.py — _should_skip_due_to_memo() (line ~865), _turn_on_switch() (line ~737), _turn_off_switch() (line ~810)

2. The Bug: Two Paths to Stale Memo

There are two distinct scenarios where the memo becomes stale, both leading to the same symptom: lights won't turn on when motion is detected.

Path A: Unverified OFF Command

The OFF command succeeds at the device level but verification fails (common with Matter devices that use the dual-command strategy). Memo stays at 'on'.

1. Motion Active
App turns on lights. Command verified. Memo = {'state': 'on', 'source': 'app'}
2. Timeout Fires (no motion)
master() calls _control_lights('off')
3. OFF Command Sent
Device turns off physically, but verification fails (Matter device, network hiccup, etc.)
4. Memo NOT Updated
_turn_off_switch() returns False (unverified). Memo still = {'state': 'on', 'source': 'app'}
5. Motion Active Again
master(motion_active_event=True) calls _control_lights('on')
6. BUG: Skip!
_should_skip_due_to_memo('on') sees memo='on' == action='on' → SKIP. Lights stay off.

Path B: External State Change

Lights are turned off by something external (Hubitat rule, physical switch, another app). The cache updates via webhook, but memo does not update for regular switches.

1. Motion Active
App turns on lights. Verified. Memo = {'state': 'on', 'source': 'app'}
2. External OFF
User turns off from Hubitat dashboard, or another automation fires. Webhook arrives → cache updated to 'off'. But _handle_switch() only updates memo for keep_off/keep_on devices, not regular switches. Memo still = {'state': 'on', 'source': 'app'}
3. Timeout Fires
_turn_off_switch() checks cache → device already 'off' → returns False (no command needed, no memo update). Memo still = {'state': 'on', 'source': 'app'}
4. Motion Active Again
_control_lights('on')
5. BUG: Skip!
_should_skip_due_to_memo('on') → memo='on' == action='on' → SKIP. Lights stay off.

4. The Fix: Cross-Check Memo Against Device State

Solution: When _should_skip_due_to_memo() says "skip," cross-check the cached device state. If the device is in the opposite state from what we'd skip for, the memo is stale — clear it and proceed with the command.

Before vs. After

Before (Bug)

# In _control_lights(): if self._should_skip_due_to_memo(device_name, action): continue # ← Blindly trusts memo
→

After (Fix)

# In _control_lights(): if self._should_skip_due_to_memo(device_name, action): if device: actual = device.get('attributes', {}).get('switch') if actual is not None and actual != action: # Memo is stale — device contradicts memo self._memoization['switch_state'].pop( device_name, None) memo_dirty = True else: continue # Memo matches reality else: continue # No device info, trust memo

5. Complete Scenario: Fixed Behavior

1. Motion Active
App turns on lights. Verified. Memo = {'state': 'on', 'source': 'app'}
2. Timeout Fires
OFF command sent. NOT verified (or device already off). Memo stays {'state': 'on', 'source': 'app'} (stale). But cache updated to 'off' by webhook.
3. Motion Active Again
_control_lights('on') called.
4. FIX: Stale Memo Detected
_should_skip_due_to_memo('on') returns True (memo='on'). But cross-check: cache says device='off'. 'off' != 'on' → memo is stale! Clear memo entry, proceed.
5. ON Command Sent
_turn_on_switch() sends ON. Verified. Memo updated = {'state': 'on', 'source': 'app'}. Lights turn on.

6. Cross-Check Truth Table

Memo State Action Device State (Cache) Result Reason
'on' 'on' 'on' SKIP Memo and device agree: already on, no action needed
'on' 'on' 'off' PROCEED Memo stale: claims 'on' but device is 'off'. Clear and send ON.
'off' 'off' 'off' SKIP Memo and device agree: already off, no action needed
'off' 'off' 'on' PROCEED Memo stale: claims 'off' but device is 'on'. Clear and send OFF.
'on' 'off' (any) PROCEED Memo != action, skip check not triggered (normal path)
'off' 'on' (any) PROCEED Memo != action, skip check not triggered (normal path)
None (any) (any) PROCEED No memo entry — always proceed

7. Why the Cache Is Trustworthy

The cross-check relies on the device cache being more current than the memo. This holds because the cache has three independent update paths, while memo only updates on verified app commands.

Update Source Cache Memo Latency
Webhook (device event) Updated NOT updated (regular switches) ~100ms
DeviceCommander (verified) Updated Updated ~1-3s
DeviceCommander (unverified) Updated (via subsequent webhook) NOT updated ~100ms (webhook)
Background refresh Updated NOT updated ~120s cycle
External change (user/rule) Updated (via webhook) NOT updated ~100ms
Design principle: The memo is intentionally conservative (verified-only updates) to prevent phantom overrides. The cross-check compensates by using the cache as a reality check when the memo would cause incorrect behavior.

8. Interaction with Other Memoization Features

Keep Devices (Always Off / Always On)

Keep devices use keep:{device_id} keys and are managed by _enforce_keep_switches(). They are excluded from _control_lights() entirely, so the stale memo fix does not affect them. Keep devices have their own source-tracking override detection.

Mode Changes

Mode changes call _reset_memoization(), which clears all memo entries. This is an independent reset that would also clear a stale memo — but only on mode change, not on motion events.

Pause / Resume

Both pause and resume reset memoization. The stale memo fix is relevant for normal runtime between mode/pause events.
