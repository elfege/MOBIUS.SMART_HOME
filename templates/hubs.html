Maker API — what it is and when to disable it

Old architecture (until 2026-05-16): the app received all device events as HTTP POSTs from each Hubitat hub's Maker API. Required per-device opt-in on the hub; brittle across firmware updates; per-event HTTP latency.

New architecture: a persistent WebSocket (ws://hub/eventsocket) streams every event from every device, no opt-in, no HTTP. Mobius writes them all into its own database (event_log + raw_events) which becomes the source of truth for state.

What Maker API still does (for now):

  • The reconcile poll pulls /devices/all every 60s as a safety net for events the WS may have missed during a reconnect window.
  • Outbound commands (turn on, set level) go through /devices/<id>/<cmd>.

Disabling it: the toggle switches both paths to Hubitat's admin API (what the web UI itself uses). Reconcile becomes per-subscribed-device /device/fullJson/<id> lookups; commands use /device/<id>/<cmd> on the admin endpoint. The admin command path is not yet production-verified — keep Maker API enabled for now unless you're actively testing the cutover.

If the eventsocket fails (hub reboot, firmware update), the landing page shows a warning and recommends re-enabling Maker API as the fallback intake until the WS is healthy again.
